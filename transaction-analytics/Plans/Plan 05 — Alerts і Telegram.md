# Plan 05 — Попередження + доставка в Telegram

## Мета
Система сама повідомляє про важливе: перевищення бюджету, аномалії, нові підписки —
миттєво в Telegram; плюс періодичний дайджест.

## Scope
- In: notifier-абстракція, Telegram-канал, rule-based алерти, weekly/monthly дайджест.
- Out: push/email; двосторонній бот (команди боту — окремо, пізніше).

## Модель даних
- `AlertLog(id, type, dedupeKey UNIQUE, payload jsonb, sentAt)` — ідемпотентність алертів.

## Кроки
1. `Notifier` interface + `TelegramNotifier` (bot token + chatId лише з env — NR3) +
   `NullNotifier` для тестів/локалки. Нове — збоку, як Sheets-підписник (NR1).
2. Правила (кожне = окремий клас з єдиним контрактом):
   - бюджет ≥80% і ≥100% (по одному разу на місяць на бюджет — dedupeKey);
   - аномальна витрата: сума > mean + 2σ по категорії за 90 днів (мін. 10 спостережень);
   - велика транзакція: > поріг з env;
   - нова підписка (з [[Plan 02 — Гігієна даних|RecurringPayment]]);
   - подвійне списання: той самий мерчант+сума в межах 5 хв.
3. Тригери: підписка на `transaction.created` (миттєві) + cron після щоденного синку (бюджети).
4. Дайджест (cron, weekly + monthly): зведення з [[Plan 03 — Агрегації]] +
   [[Plan 04 — Бюджети й цілі]] у людському форматі.

## Критерії приймання
- [ ] Кожен алерт надсилається **один раз** (повторний прогін правила → 0 повідомлень;
      тест на dedupeKey).
- [ ] Падіння Telegram не ламає синк (ізоляція, як у Sheets — [[Events & Export]]).
- [ ] Секрети відсутні в коді й БД (NR3); без token — NullNotifier, все працює.
- [ ] Кожне правило покрито unit-тестом (спрацьовує/не спрацьовує на фікстурах).
- [ ] Дайджест формується з агрегатів і містить: cash flow, топ категорій, статус бюджетів.
- [ ] Інтеграційний тест з mock notifier; `tsc` чистий.
