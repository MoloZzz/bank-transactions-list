# Plan 02 — Гігієна даних (перекази, курси, підписки)

## Мета
Статистика не бреше: власні перекази не є витратами, все зводиться до UAH,
регулярні платежі відомі системі.

## Scope
- In: детекція внутрішніх переказів, сервіс курсів НБУ + кеш, детекція підписок.
- Out: курси комерційних банків; прогнозування підписок наперед (лише факт-детекція).

## Модель даних
- `Transaction`: + `isInternalTransfer boolean`, `linkedTransactionId?`.
- `ExchangeRate(date, currency, rateMinorScaled)` — кеш НБУ, UNIQUE(date, currency).
- `RecurringPayment(id, normalizedMerchant, expectedAmountMinor, currency, periodDays,
  lastSeenAt, active)`.

## Кроки
1. **Перекази**: job шукає пари (різні власні рахунки, протилежний знак, |сума| збігається
   з допуском на конвертацію, час ±24 год) → позначає обидві + лінкує. Ручний unlink/link.
2. **НБУ**: клієнт API НБУ (курс на дату), кеш у `ExchangeRate`, 1 запит на (дата, валюта).
   Хелпер `toBaseUah(amountMinor, currency, date)`. Цей сервіс використовує і крок 6
   roadmap (estimate, `rateSource=NBU`).
3. **Підписки**: job групує за normalizedMerchant, шукає ≥3 списання з періодом
   28–33 дні (або 6–8/89–95) і сумою ±10% → створює/оновлює `RecurringPayment`;
   деактивація, якщо 2 періоди пропущено.

## Критерії приймання
- [ ] Переказ між власними картками (у т.ч. крос-валютний UAH→USD) позначено; у
      агрегаціях витрат він відсутній (тест-кейс на пару).
- [ ] Несиметрична пара (реальна витрата) НЕ позначається (негативний тест).
- [ ] Курс НБУ на дату повертається з кешу при повторному запиті (mock http → 1 виклик).
- [ ] `toBaseUah` — цілі мінорні одиниці, без float ([[Invariants]] #1).
- [ ] Підписка з 3+ повторами детектиться на фікстурах; одноразові покупки — ні.
- [ ] Unit + інтеграційний тести, `tsc` чистий.
